# 2026-06-04 Qwen3.5 MTP Local Model Bring-up

## Goal

Evaluate whether `unsloth/Qwen3.5-2B-MTP-GGUF` can replace or supplement the
current Gemma local model for Hermes-agent while keeping a large context window.

## Changes

- Added Qwen download presets to `scripts/download_model.sh`.
- Updated `scripts/env.sh` so explicit environment variables and profile values
  override `.env`.
- Added `model-profiles/qwen3.5-2b-mtp-q4-xl.env`.
- Added `scripts/run_model_profile.sh` as the model switching entry point.
- Added `docker-compose.gpu-mtp.example.yml` for MTP, flash attention, and KV
  cache type flags.

## Runtime Notes

- `LLAMA_CTX_SIZE=130000` is feasible for this Qwen 2B Q4 XL model on the
  tested 6GB GPU.
- First short chat request at 130000 ctx returned `ready`.
- First prompt processing was slow, about 92 seconds for 25 prompt tokens.
- Generation after prompt processing was about 41 tokens/sec.
- Before adding the MTP compose override, llama.cpp logged:

```text
speculative decoding will use checkpoints
no implementations specified for speculative decoding
```

That means the MTP-capable model was running, but MTP speculative decoding was
not actually enabled during the initial context sweep.

## MTP Override Verification

After adding `docker-compose.gpu-mtp.example.yml`, the same profile started at
`LLAMA_CTX_SIZE=130000` and llama.cpp reported:

```text
creating MTP draft context against the target model
adding speculative implementation 'draft-mtp'
speculative decoding context initialized
```

Observed VRAM with MTP enabled at `n_ctx=130048`:

```text
~5.26 GiB / 6 GiB
```

Short chat verification returned `ready`. The short response is not a useful
speed benchmark because speculative setup overhead dominated:

```text
prompt time: ~99 seconds for 25 tokens
generation: ~0.13 tokens/sec for 2 output tokens
draft_n: 6
draft_n_accepted: 4
```

Next verification should use a longer generation task and compare MTP on/off
with the same prompt.
